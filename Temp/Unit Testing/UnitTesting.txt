================================================================
              UNIT TESTING — JUnit 5 + Mockito
================================================================

WHAT IS UNIT TESTING?
--------------------------
Unit Testing is the practice of testing individual units of
code (methods, classes) in isolation to ensure they work
correctly. A "unit" is the smallest testable piece of code.

Goals:
  - Verify each unit of code behaves as expected.
  - Catch bugs early (before integration or production).
  - Provide living documentation of the code.
  - Enable safe refactoring (tests guard against regression).
  - Drive better design (testable code = modular code).

Testing Pyramid:
                   /\
                  /  \
                 / E2E \         <- Few, slow, expensive
                /--------\
               /Integration\     <- Some, moderate cost
              /--------------\
             /   Unit Tests   \   <- Many, fast, cheap
            /------------------\


================================================================
TESTING TERMINOLOGY
================================================================

  Term               | Description
  -------------------|------------------------------------------------
  Unit Test          | Tests a single class/method in isolation
  Integration Test   | Tests multiple components working together
  E2E Test           | Tests entire application flow
  Test Suite         | Collection of test classes/methods
  Assertion          | Statement verifying expected outcome
  Mock               | Fake object simulating a dependency
  Stub               | Simplified implementation returning fixed data
  Spy                | Real object with some methods overridden
  Fixture            | Fixed state/data for tests
  SUT                | System Under Test (the class being tested)
  AAA Pattern        | Arrange → Act → Assert
  GWT Pattern        | Given → When → Then (BDD style)
  Code Coverage      | % of production code exercised by tests
  Test Double        | Generic term for Mock/Stub/Spy/Fake/Dummy

JUNIT 5 — CORE ANNOTATIONS
================================================================

  Annotation              | Description
  ------------------------|-------------------------------------------
  @Test                   | Marks a method as a test
  @BeforeEach             | Runs before EACH test method
  @AfterEach              | Runs after EACH test method
  @BeforeAll              | Runs ONCE before all tests (static)
  @AfterAll               | Runs ONCE after all tests (static)
  @DisplayName            | Custom name for test in report
  @Disabled               | Skip a test or class
  @Nested                 | Group related tests in inner classes
  @Tag                    | Label tests for filtering (e.g. "slow")
  @RepeatedTest(n)        | Run same test n times
  @ParameterizedTest      | Run test with multiple inputs
  @ValueSource            | Array of values for parameterized test
  @CsvSource              | CSV pairs for parameterized test
  @MethodSource           | Stream/method as test data source
  @EnumSource             | Enum values as test data
  @TempDir                | Inject temporary directory
  @TestMethodOrder        | Control test execution order
  @Timeout                | Fail if test exceeds time limit
  @ExtendWith             | Register extensions (e.g., Mockito)


================================================================
JUNIT 5 — ASSERTIONS
================================================================

import static org.junit.jupiter.api.Assertions.*;

BASIC ASSERTIONS:
  assertEquals(expected, actual)
  assertEquals(expected, actual, "custom message")
  assertNotEquals(unexpected, actual)
  assertTrue(condition)
  assertFalse(condition)
  assertNull(object)
  assertNotNull(object)
  assertSame(expected, actual)       // same reference (==)
  assertNotSame(expected, actual)

ARRAY / COLLECTION:
  assertArrayEquals(expectedArr, actualArr)
  assertIterableEquals(expectedList, actualList)

EXCEPTION TESTING:
  Exception ex = assertThrows(
      IllegalArgumentException.class,
      () -> service.method(null)
  );
  assertEquals("message", ex.getMessage());

  assertDoesNotThrow(() -> service.method("valid"));

GROUPED ASSERTIONS (all run even if one fails):
  assertAll(
      () -> assertEquals("Alice", user.getName()),
      () -> assertEquals(30,      user.getAge()),
      () -> assertEquals("NY",    user.getCity())
  );

TIMEOUT:
  assertTimeout(Duration.ofMillis(100), () -> service.fast());
  assertTimeoutPreemptively(Duration.ofMillis(100), () -> ...);

FAIL EXPLICITLY:
  fail("Should not reach here");


================================================================
JUNIT 5 — ASSUMPTIONS
================================================================
Skip a test if condition is not met (not a failure).

  import static org.junit.jupiter.api.Assumptions.*;

  assumeTrue(System.getenv("CI") != null);
  assumeFalse(isWindows);
  assumingThat(isProduction, () -> assertEquals(...));


================================================================
JUNIT 5 — PARAMETERIZED TESTS
================================================================

@ParameterizedTest
@ValueSource(ints = {1, 2, 3, 4, 5})
void testPositive(int number) {
    assertTrue(number > 0);
}

@ParameterizedTest
@ValueSource(strings = {"", " ", "\t", "\n"})
void testBlankStrings(String input) {
    assertTrue(input.isBlank());
}

@ParameterizedTest
@CsvSource({
    "Alice, 30, NY",
    "Bob,   25, LA",
    "Carol, 35, SF"
})
void testUser(String name, int age, String city) {
    User user = new User(name, age, city);
    assertNotNull(user);
    assertEquals(name, user.getName());
}

@ParameterizedTest
@CsvFileSource(resources = "/test-data.csv", numLinesToSkip = 1)
void testFromFile(String input, String expected) {
    assertEquals(expected, service.process(input));
}

@ParameterizedTest
@MethodSource("provideStrings")
void testWithMethodSource(String input, boolean expected) {
    assertEquals(expected, StringUtils.isPalindrome(input));
}

static Stream<Arguments> provideStrings() {
    return Stream.of(
        Arguments.of("racecar", true),
        Arguments.of("hello",   false),
        Arguments.of("madam",   true)
    );
}

@ParameterizedTest
@EnumSource(value = Status.class, names = {"ACTIVE", "PENDING"})
void testActiveStatuses(Status status) {
    assertTrue(service.isActive(status));
}


================================================================
JUNIT 5 — LIFECYCLE
================================================================

@BeforeAll   (static)
    |
    v
@BeforeEach
    |
    v
@Test  ← runs once per test
    |
    v
@AfterEach
    |
    v (repeat for each test)
    |
    v
@AfterAll    (static)

Example:
  @BeforeAll
  static void setupDatabase() {
      // runs once — connect to test DB
  }

  @BeforeEach
  void setup() {
      // runs before each test — reset state
      userService = new UserService(mockRepo);
  }

  @AfterEach
  void teardown() {
      // runs after each test — cleanup
  }

  @AfterAll
  static void closeDatabase() {
      // runs once — disconnect
  }

MOCKITO — OVERVIEW
================================================================

Mockito is a mocking framework for Java unit tests.
It creates fake (mock) objects to isolate the unit under test.

KEY CONCEPTS:
  Mock  : Fake object — all methods return default values.
  Stub  : Configured mock — methods return specified values.
  Spy   : Real object — only specific methods overridden.
  Verify: Assert that a method was called as expected.
  Captor: Capture arguments passed to mock methods.

ENABLE MOCKITO IN JUNIT 5:
  @ExtendWith(MockitoExtension.class)
  class OrderServiceTest { ... }


================================================================
MOCKITO — CREATING MOCKS
================================================================

  // Option 1: Annotation (preferred)
  @Mock
  private OrderRepository orderRepository;

  @InjectMocks
  private OrderService orderService;   // injects mocks automatically

  // Option 2: Programmatic
  OrderRepository mock = Mockito.mock(OrderRepository.class);

  // Option 3: Spy (real object, partial mock)
  @Spy
  private List<String> spyList = new ArrayList<>();

  // Option 4: Static mock (Mockito 3.4+)
  try (MockedStatic<UUID> mockedUUID = Mockito.mockStatic(UUID.class)) {
      mockedUUID.when(UUID::randomUUID).thenReturn(fixedUUID);
      // ...
  }


================================================================
MOCKITO — STUBBING (WHEN / THEN)
================================================================

BASIC STUBBING:
  when(repo.findById(1L)).thenReturn(Optional.of(user));
  when(repo.findAll()).thenReturn(List.of(user1, user2));
  when(repo.save(any())).thenAnswer(i -> i.getArgument(0));

RETURN MULTIPLE VALUES (sequential):
  when(service.getValue())
      .thenReturn(1)
      .thenReturn(2)
      .thenReturn(3);
  // 1st call=1, 2nd call=2, 3rd+ call=3

THROW EXCEPTION:
  when(repo.findById(99L))
      .thenThrow(new ResourceNotFoundException("Not found"));

  doThrow(new RuntimeException("DB error"))
      .when(repo).deleteById(1L);

VOID METHODS:
  doNothing().when(emailService).sendEmail(anyString());
  doThrow(new RuntimeException()).when(emailService).sendEmail(anyString());

CUSTOM ANSWER:
  when(repo.save(any(Order.class))).thenAnswer(invocation -> {
      Order order = invocation.getArgument(0);
      order.setId(100L);
      return order;
  });

ARGUMENT MATCHERS:
  any()               any object (including null)
  any(Type.class)     any object of given type
  anyString()         any non-null String
  anyInt()            any int
  anyLong()           any long
  anyList()           any List
  eq(value)           exact value match
  isNull()            null argument
  isNotNull()         non-null argument
  startsWith("pre")   String starting with prefix
  endsWith("suf")     String ending with suffix
  contains("text")    String containing text
  matches("regex")    String matching regex
  argThat(predicate)  custom predicate match

  NOTE: If you use matchers, ALL arguments must use matchers.
  when(repo.findByNameAndAge(eq("Alice"), anyInt())).thenReturn(...);


================================================================
MOCKITO — VERIFICATION
================================================================

BASIC VERIFY:
  verify(repo).findById(1L);
  verify(repo, times(2)).save(any());
  verify(repo, never()).deleteById(anyLong());
  verify(repo, atLeast(1)).findAll();
  verify(repo, atMost(3)).findAll();
  verify(repo, atLeastOnce()).findById(any());
  verify(repo, only()).findById(1L);    // only this method called

VERIFY ORDER:
  InOrder inOrder = inOrder(repo, emailService);
  inOrder.verify(repo).save(order);
  inOrder.verify(emailService).sendEmail(anyString());

VERIFY NO MORE INTERACTIONS:
  verifyNoMoreInteractions(repo);
  verifyNoInteractions(emailService);

ARGUMENT CAPTOR:
  @Captor
  ArgumentCaptor<Order> orderCaptor;

  verify(repo).save(orderCaptor.capture());
  Order captured = orderCaptor.getValue();
  assertEquals("Laptop", captured.getProduct());

  // Capture multiple calls
  verify(emailService, times(2)).send(captor.capture());
  List<String> values = captor.getAllValues();


================================================================
MOCKITO — SPY
================================================================

Spy wraps a REAL object. Only overridden methods are mocked.

  @Spy
  private List<String> spyList = new ArrayList<>();

  spyList.add("Apple");   // real method called
  spyList.add("Banana");

  doReturn(100).when(spyList).size();  // override size()
  assertEquals(100, spyList.size());   // uses stub
  assertEquals("Apple", spyList.get(0)); // uses real

  verify(spyList).add("Apple");

  NOTE: Use doReturn() with spies (not when().thenReturn())
        to avoid calling the real method during stubbing.


================================================================
MOCKITO — BDD STYLE (given/when/then)
================================================================

  import static org.mockito.BDDMockito.*;

  // given
  given(repo.findById(1L)).willReturn(Optional.of(user));

  // when
  User result = service.findById(1L);

  // then
  assertNotNull(result);
  then(repo).should().findById(1L);
  then(emailService).shouldHaveNoInteractions();


================================================================
WRITING GOOD UNIT TESTS — AAA PATTERN
================================================================

  @Test
  @DisplayName("Should return user when valid ID provided")
  void shouldReturnUserWhenValidIdProvided() {
      // ARRANGE
      Long userId = 1L;
      User expected = new User(userId, "Alice", "alice@example.com");
      when(userRepository.findById(userId))
          .thenReturn(Optional.of(expected));

      // ACT
      User actual = userService.findById(userId);

      // ASSERT
      assertNotNull(actual);
      assertEquals(expected.getId(),    actual.getId());
      assertEquals(expected.getName(),  actual.getName());
      assertEquals(expected.getEmail(), actual.getEmail());
      verify(userRepository, times(1)).findById(userId);
  }


================================================================
TESTING EXCEPTIONS
================================================================

  @Test
  @DisplayName("Should throw exception when user not found")
  void shouldThrowWhenUserNotFound() {
      // Arrange
      when(repo.findById(99L)).thenReturn(Optional.empty());

      // Act + Assert
      ResourceNotFoundException ex = assertThrows(
          ResourceNotFoundException.class,
          () -> userService.findById(99L)
      );

      assertEquals("User not found with id: 99", ex.getMessage());
      verify(repo).findById(99L);
  }


================================================================
TESTING VOID METHODS
================================================================

  @Test
  void shouldDeleteUserSuccessfully() {
      // Arrange
      when(repo.existsById(1L)).thenReturn(true);
      doNothing().when(repo).deleteById(1L);

      // Act
      assertDoesNotThrow(() -> userService.deleteUser(1L));

      // Assert
      verify(repo).existsById(1L);
      verify(repo).deleteById(1L);
  }

  @Test
  void shouldThrowWhenDeletingNonExistentUser() {
      when(repo.existsById(99L)).thenReturn(false);

      assertThrows(ResourceNotFoundException.class,
          () -> userService.deleteUser(99L));

      verify(repo, never()).deleteById(any());
  }

SPRING BOOT TESTING ANNOTATIONS
================================================================

  Annotation           | Description
  ---------------------|------------------------------------------
  @SpringBootTest      | Loads full application context
  @WebMvcTest          | Loads only MVC layer (Controller)
  @DataJpaTest         | Loads only JPA/repository layer
  @JsonTest            | Tests JSON serialization/deserialization

  @SpyBean             | Creates and injects Mockito spy into context
  @AutoConfigureMockMvc| Auto-configure MockMvc
  @Transactional       | Rollback DB changes after each test
  @Sql                 | Execute SQL scripts before/after test
  @ActiveProfiles      | Activate specific profile for tests

CONTROLLER TEST WITH MockMvc:
  @WebMvcTest(UserController.class)
  class UserControllerTest {

      @Autowired MockMvc mockMvc;
      @MockBean  UserService userService;

      @Test
      void shouldGetUser() throws Exception {
          when(userService.findById(1L))
              .thenReturn(new User(1L, "Alice", "a@b.com"));

          mockMvc.perform(get("/api/users/1")
                  .contentType(MediaType.APPLICATION_JSON))
              .andExpect(status().isOk())
              .andExpect(jsonPath("$.name").value("Alice"))
              .andExpect(jsonPath("$.email").value("a@b.com"));
      }

      @Test
      void shouldReturn404WhenNotFound() throws Exception {
          when(userService.findById(99L))
              .thenThrow(new ResourceNotFoundException("Not found"));

          mockMvc.perform(get("/api/users/99"))
              .andExpect(status().isNotFound());
      }
  }

REPOSITORY TEST WITH @DataJpaTest:
  @DataJpaTest
  class UserRepositoryTest {

      @Autowired UserRepository repo;
      @Autowired TestEntityManager em;

      @Test
      void shouldFindByEmail() {
          em.persist(new User(null, "Alice", "alice@example.com"));
          em.flush();

          Optional<User> found = repo.findByEmail("alice@example.com");
          assertTrue(found.isPresent());
          assertEquals("Alice", found.get().getName());
      }
  }

CODE COVERAGE
================================================================

WHAT IS CODE COVERAGE?
  Measures how much of production code is exercised by tests.

TYPES:
  Line Coverage    : % of lines executed.
  Branch Coverage  : % of branches (if/else) executed.
  Method Coverage  : % of methods called.
  Class Coverage   : % of classes instantiated.

TOOLS:
  JaCoCo  : Most popular Java coverage tool.
  IntelliJ: Built-in coverage runner.
  SonarQube: Code quality + coverage dashboard.

MAVEN JACOCO SETUP:
  <plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.11</version>
    <executions>
      <execution>
        <goals><goal>prepare-agent</goal></goals>
      </execution>
      <execution>
        <id>report</id>
        <phase>test</phase>
        <goals><goal>report</goal></goals>
      </execution>
    </executions>
  </plugin>

Run: mvn test   → generates report in target/site/jacoco/

TARGET COVERAGE:
  70%+ line coverage is a common minimum.
  80%+ is good for critical business logic.
  100% is rarely practical or necessary.
  Coverage is a metric — not a guarantee of quality.

TEST NAMING CONVENTIONS
================================================================

FORMATS:
  1. methodName_StateUnderTest_ExpectedBehaviour
     findById_ExistingId_ReturnsUser
     findById_NonExistingId_ThrowsException

  2. should_ExpectedBehaviour_When_StateUnderTest
     should_ReturnUser_When_ValidIdProvided
     should_ThrowException_When_UserNotFound

  3. Plain English (with @DisplayName)
     @DisplayName("Should return all active users")

RULE: Test names should read like a specification.


================================================================
BEST PRACTICES
================================================================

  1. ONE ASSERTION PER TEST (ideally)
     Each test verifies one behavior. Easier to identify failures.

  2. ARRANGE-ACT-ASSERT (AAA)
     Clear structure for every test method.

  3. INDEPENDENT TESTS
     Tests must not depend on each other or execution order.
     Reset state in @BeforeEach.

  4. FAST TESTS
     Unit tests should run in milliseconds.
     Avoid real DB, network, or file I/O in unit tests.

  5. DETERMINISTIC
     Same test must produce same result every time.
     Avoid random data or time-dependent logic (mock clocks).

  6. TEST BEHAVIOUR, NOT IMPLEMENTATION
     Test what a method does, not how it does it internally.
     Avoids brittle tests that break on refactoring.

  7. AVOID TESTING PRIVATE METHODS
     Test through public API. Private methods are tested
     indirectly via public method tests.

  8. MEANINGFUL ASSERTIONS
     Always assert something useful. Avoid empty tests.

  9. TEST EDGE CASES
     Null inputs, empty collections, boundary values,
     negative numbers, maximum values.

 10. MOCK ONLY EXTERNAL DEPENDENCIES
     Don't mock the class you're testing (SUT).
     Mock only: repositories, external services, third-party APIs.

 11. KEEP TESTS CLOSE TO SOURCE
     Test class in same package as source class.
     UserService.java -> UserServiceTest.java

 12. USE @DisplayName
     Descriptive names make test reports readable.


================================================================
COMMON MISTAKES TO AVOID
================================================================

  - Testing implementation details (internal method calls).
  - Over-mocking (mocking everything, including value objects).
  - Not testing failure/exception paths.
  - Ignoring flaky tests (random failures hide real bugs).
  - Writing tests after code (test-driven development is better).
  - Asserting mock interactions when not meaningful.
  - Using Thread.sleep() in tests (use mocked clocks).
  - Sharing mutable state between tests.
  - Massive test methods testing multiple behaviors.


================================================================
TDD — TEST DRIVEN DEVELOPMENT
================================================================

CYCLE: Red → Green → Refactor

  Red    : Write a FAILING test for new functionality.
  Green  : Write MINIMUM code to make the test pass.
  Refactor: Clean up code without breaking tests.

BENEFITS:
  - Forces modular, testable design.
  - Tests act as specification.
  - Ensures 100% test coverage of new code.
  - Reduces debugging time significantly.

EXAMPLE:
  // 1. RED — write failing test
  @Test
  void shouldCalculateTax() {
      assertEquals(20.0, taxService.calculate(100.0, 0.20));
  }

  // 2. GREEN — minimal implementation
  double calculate(double amount, double rate) {
      return amount * rate;
  }

  // 3. REFACTOR — improve code, keep tests green
  double calculate(double amount, double taxRate) {
      if (amount < 0) throw new IllegalArgumentException("Negative amount");
      return amount * taxRate;
  }
