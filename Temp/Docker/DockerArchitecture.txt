                   DOCKER ARCHITECTURE
================================================================

WHAT IS DOCKER?
--------------------------
Docker is an open-source platform that enables developers to
build, ship, and run applications inside lightweight, portable
containers. A container packages the application code along
with all its dependencies, libraries, and configuration so it
runs consistently across any environment (dev, test, prod).

Docker uses OS-level virtualization instead of full virtual
machines, making containers much lighter and faster to start.

CORE COMPONENTS
--------------------------
1. DOCKER ENGINE
   - The core runtime that builds and runs containers.
   - Consists of:
     a) Docker Daemon (dockerd) — background service managing
        containers, images, volumes, networks.
     b) Docker CLI (docker) — command-line tool to interact
        with the daemon.
     c) REST API — interface between CLI and daemon.

2. DOCKER IMAGE
   - A read-only template used to create containers.
   - Built from a Dockerfile using layered filesystem.
   - Stored in a registry (Docker Hub, ECR, GCR, etc.).
   - Each layer represents one instruction in Dockerfile.

3. DOCKER CONTAINER
   - A running instance of a Docker image.
   - Has its own filesystem, network, and process space.
   - Ephemeral by default (data lost when stopped).
   - Multiple containers can run from the same image.

4. DOCKERFILE
   - A text file with instructions to build a Docker image.
   - Each instruction creates a new image layer.
   - Common instructions:
     FROM    — base image
     RUN     — execute shell command during build
     COPY    — copy files from host to image
     ADD     — like COPY but supports URLs and tar extraction
     WORKDIR — set working directory
     ENV     — set environment variable
     EXPOSE  — document which port the app uses
     CMD     — default command when container starts
     ENTRYPOINT — fixed executable for the container

5. DOCKER REGISTRY
   - A storage and distribution system for Docker images.
   - Docker Hub  : public registry (hub.docker.com)
   - ECR         : AWS Elastic Container Registry
   - GCR         : Google Container Registry
   - ACR         : Azure Container Registry
   - Private     : self-hosted registry

6. DOCKER VOLUME
   - Persistent storage that survives container restarts.
   - Stored on host filesystem managed by Docker.
   - Types:
     a) Named Volume   : docker volume create mydata
     b) Bind Mount     : maps host directory to container
     c) tmpfs Mount    : stored in memory only

7. DOCKER NETWORK
   - Enables communication between containers.
   - Types:
     a) bridge   — default; containers on same host
     b) host     — container shares host network stack
     c) none     — no networking
     d) overlay  — multi-host networking (Docker Swarm)
     e) macvlan  — assign MAC address to container

8. DOCKER COMPOSE
   - Tool to define and run multi-container applications.
   - Uses a YAML file (docker-compose.yml).
   - Manages services, networks, and volumes together.

9. DOCKER HUB
   - The default public registry for Docker images.
   - Hosts official images: nginx, postgres, redis, etc.

HOW DOCKER WORKS (FLOW)
--------------------------
Step 1: Write a Dockerfile describing the app environment.
Step 2: Build an image: docker build -t myapp:1.0 .
Step 3: Push image to registry: docker push myapp:1.0
Step 4: Pull image on any machine: docker pull myapp:1.0
Step 5: Run a container: docker run -p 8080:8080 myapp:1.0
Step 6: Container starts with all dependencies included.


DOCKERFILE EXAMPLE (Java Spring Boot App)
--------------------------
  # Stage 1: Build
  FROM maven:3.9-eclipse-temurin-17 AS builder
  WORKDIR /app
  COPY pom.xml .
  COPY src ./src
  RUN mvn clean package -DskipTests

  # Stage 2: Run
  FROM eclipse-temurin:17-jre-alpine
  WORKDIR /app
  COPY --from=builder /app/target/myapp.jar app.jar
  EXPOSE 8080
  ENV JAVA_OPTS="-Xms256m -Xmx512m"
  ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]


DOCKER COMPOSE EXAMPLE
--------------------------
  version: '3.9'
  services:
    app:
      build: .
      ports:
        - "8080:8080"
      environment:
        - SPRING_DATASOURCE_URL=jdbc:postgresql://db:5432/mydb
      depends_on:
        - db
      networks:
        - app-network

    db:
      image: postgres:15
      environment:
        POSTGRES_DB: mydb
        POSTGRES_USER: user
        POSTGRES_PASSWORD: password
      volumes:
        - pgdata:/var/lib/postgresql/data
      networks:
        - app-network

  volumes:
    pgdata:

  networks:
    app-network:


ESSENTIAL DOCKER COMMANDS
--------------------------
IMAGE COMMANDS:
  docker build -t name:tag .       Build image from Dockerfile
  docker images                    List all local images
  docker pull image:tag            Download image from registry
  docker push image:tag            Upload image to registry
  docker rmi image:tag             Remove image
  docker tag source target         Tag an image

CONTAINER COMMANDS:
  docker run -d -p 8080:80 nginx   Run container in background
  docker run -it ubuntu bash       Run interactive container
  docker ps                        List running containers
  docker ps -a                     List all containers
  docker stop container_id         Stop a container
  docker start container_id        Start a stopped container
  docker restart container_id      Restart a container
  docker rm container_id           Remove a container
  docker logs container_id         View container logs
  docker exec -it container bash   Open shell in container
  docker inspect container_id      Detailed container info
  docker stats                     Live resource usage

VOLUME COMMANDS:
  docker volume create mydata      Create named volume
  docker volume ls                 List volumes
  docker volume rm mydata          Remove volume
  docker volume inspect mydata     Volume details

NETWORK COMMANDS:
  docker network create mynet      Create network
  docker network ls                List networks
  docker network inspect mynet     Network details
  docker network connect mynet c1  Connect container to network

COMPOSE COMMANDS:
  docker compose up -d             Start all services
  docker compose down              Stop and remove containers
  docker compose logs -f           Follow logs
  docker compose ps                List service containers
  docker compose build             Rebuild images
  docker compose scale app=3       Scale service to 3 instances

SYSTEM COMMANDS:
  docker system prune              Remove unused resources
  docker system df                 Show disk usage


DOCKER IMAGE LAYERS
--------------------------
Each instruction in a Dockerfile adds a new read-only layer.
Layers are cached — unchanged layers are reused on rebuild.

  FROM ubuntu:22.04        <-- Layer 1 (base)
  RUN apt-get update       <-- Layer 2
  RUN apt-get install -y java  <-- Layer 3
  COPY app.jar /app/       <-- Layer 4
  CMD ["java", "-jar", "/app/app.jar"]  <-- Layer 5 (writable)

Benefit: If only app.jar changes, only Layer 4+ is rebuilt.


MULTI-STAGE BUILDS
--------------------------
Reduces final image size by separating build and runtime stages.

  Stage 1 (builder): Full JDK + Maven + source code
  Stage 2 (runtime): Only JRE + compiled JAR

Result: Final image is much smaller (no build tools included).


BEST PRACTICES
--------------------------
1. Use official base images (eclipse-temurin, alpine variants).
2. Use multi-stage builds to reduce image size.
3. Put frequently changing instructions (COPY src) at the end
   to maximize layer caching.
4. Use .dockerignore to exclude unnecessary files.
5. Run containers as non-root user for security.
6. Use specific image tags instead of 'latest'.
7. Keep images small — remove build artifacts and caches.
8. One process per container principle.
9. Use environment variables for configuration.
10. Use health checks to monitor container status.


COMMON USE CASES
--------------------------
- Package and ship microservices consistently
- Set up dev environments without "works on my machine" issues
- CI/CD pipelines for automated build and deployment
- Run multiple isolated services on a single host
- Blue-green deployments with zero downtime
- Scaling services with Docker Swarm or Kubernetes


DOCKER VS KUBERNETES
--------------------------
Docker     : Runs and manages containers on a single host.
Kubernetes : Orchestrates containers across multiple hosts,
             handles scaling, load balancing, self-healing,
             rolling updates, and service discovery.
             Docker is often used with Kubernetes together.
